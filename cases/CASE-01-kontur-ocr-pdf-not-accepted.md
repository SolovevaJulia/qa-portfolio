## CASE-01 — Kontur OCR: PDF не принимается в модалке распознавания паспорта (CRM)

**Контекст:**  
В CRM в инструменте **Контур (CV) → Распознавание паспорта** (модалка «Проверить физ. лицо») пользователь загружает файл первой страницы паспорта для распознавания. По бизнес-ожиданию поведение должно быть **унифицировано** с разделом **«Документы клиента»**, где PDF принимается.

**Проблема:**  
PDF (одностраничный) **не принимается**: загрузка игнорируется, распознавание не запускается. Это создаёт “тихий” отказ без понятной обратной связи.

**Что сделала:**
1. Проверила сценарий загрузки PDF в модалке «Проверить физ. лицо» и зафиксировала **стабильную воспроизводимость (5/5)**.
2. Сопоставила ожидание с уже существующим поведением в продукте: “как в **Документах клиента**”.
3. Описала дефект так, чтобы разработчику было проще принять решение:  
   либо **добавляем поддержку PDF**, либо **явно запрещаем** и показываем понятное сообщение (согласовать ТЗ).
4. Сформулировала **acceptance criteria** для фикса и ретеста (поддержка/ошибка, стабильность модалки, отсутствие JS-ошибок, единое поведение).

**Артефакты:**
- Bug report: [BUG-2025-10-30-kontur-ocr-pdf-not-accepted](bugs/crm/BUG-2025-10-30-kontur-ocr-pdf-not-accepted.md)
- Окружение: test, `v.689ce285`, admin, Windows 11 / Chrome / 1920×1080

**Результат:**  
Сформирован дефект с чёткой развилкой “поддерживаем vs запрещаем”, привязанный к бизнес-ожиданию и унификации поведения. Есть проверяемые acceptance для фикса и дальнейшего ретеста.

**Что этот кейс показывает:**
- Умение фиксировать “тихие” сбои без UI-ошибки и переводить их в понятную постановку.
- Работа с требованиями/ожиданиями бизнеса и унификация поведения в разных частях продукта.
- Формулирование проверяемых acceptance criteria под разработку и ретест.
